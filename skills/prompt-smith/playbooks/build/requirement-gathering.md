# 요구사항 수집 가이드

BUILD Mode의 GATHER 단계에서 사용하는 상세 가이드입니다.

---

## 수집 프레임워크

```
┌─────────────────────────────────────────────────────────────┐
│                   요구사항 수집 5W1H                         │
├─────────────────────────────────────────────────────────────┤
│  WHO    누가 사용하는가?                                     │
│  WHAT   무엇을 달성하려는가?                                 │
│  WHERE  어떤 맥락/환경에서?                                  │
│  WHEN   언제/얼마나 자주?                                    │
│  WHY    왜 필요한가?                                         │
│  HOW    어떻게 성공을 측정하는가?                            │
└─────────────────────────────────────────────────────────────┘
```

---

## 필수 수집 항목

### 1. 목표 (WHAT)

**질문**:
- "이 프롬프트로 무엇을 달성하고 싶으세요?"
- "최종 결과물은 무엇인가요?"

**좋은 답변 예시**:
- "고객 문의를 분류하고 우선순위를 매기고 싶어요"
- "기술 문서를 비개발자도 이해할 수 있게 요약하고 싶어요"

**나쁜 답변 (추가 질문 필요)**:
- "AI 도우미가 필요해요" → "어떤 작업을 도와줄까요?"
- "좋은 답변을 원해요" → "어떤 기준으로 '좋은'을 정의하시나요?"

---

### 2. 대상 사용자 (WHO)

**질문**:
- "누가 이 프롬프트를 사용하나요?"
- "누가 결과물을 소비하나요?"

**고려 사항**:
| 사용자 유형 | 고려할 점 |
|-------------|----------|
| 개발자 | 기술 용어 OK, 코드 예시 유용 |
| 비개발자 | 기술 용어 최소화, 시각적 설명 |
| 경영진 | 요약 중심, 결정 지원 |
| 고객 | 친절한 톤, 전문 용어 설명 |

---

### 3. 도메인 (WHERE)

**질문**:
- "어떤 분야/산업에서 사용되나요?"
- "특별한 규정이나 표준이 있나요?"

**도메인별 고려 사항**:

```markdown
## 금융
- PCI-DSS, SOX 준수
- 숫자 정확성 필수
- 법적 면책 조항

## 의료
- HIPAA 준수
- 의학적 조언 금지
- 전문가 상담 안내

## 법률
- 법적 조언 면책
- 관할권 명시
- 정확한 인용

## 교육
- 연령대 적절성
- 학습 수준 맞춤
- 격려하는 톤

## E-commerce
- 환불/반품 정책
- 재고 정보 정확성
- 개인정보 보호
```

---

### 4. 성공 기준 (HOW)

**질문**:
- "어떤 결과가 나오면 성공인가요?"
- "실패는 어떻게 정의되나요?"

**좋은 성공 기준**:
- ✅ "분류 정확도 95% 이상"
- ✅ "응답 시간 3초 이내"
- ✅ "고객 만족도 4.5/5 이상"

**나쁜 성공 기준**:
- ❌ "좋은 결과" (기준 없음)
- ❌ "빠른 응답" (정량화 없음)
- ❌ "정확한 정보" (측정 방법 없음)

---

## 권장 수집 항목

### 5. 입력 형식

**질문**:
- "어떤 데이터가 입력으로 들어오나요?"
- "입력 예시를 보여주실 수 있나요?"

**확인 사항**:
```markdown
- [ ] 데이터 형식 (텍스트/JSON/CSV/이미지)
- [ ] 데이터 크기 (짧은 문장 vs 긴 문서)
- [ ] 언어 (한국어/영어/다국어)
- [ ] 특수 문자/이모지 포함 여부
- [ ] 구조화 여부 (자유 텍스트 vs 정형 데이터)
```

---

### 6. 출력 형식

**질문**:
- "결과물이 어떤 형태여야 하나요?"
- "이 결과물을 어떻게 사용하나요?"

**형식 옵션**:

| 용도 | 권장 형식 |
|------|----------|
| API 연동 | JSON |
| 사람 읽기 | 마크다운 |
| 스프레드시트 | CSV/표 |
| 보고서 | 구조화된 마크다운 |

---

### 7. 제약 조건

**질문**:
- "반드시 피해야 할 것이 있나요?"
- "어떤 제한이 있나요?"

**제약 유형**:

```markdown
## 콘텐츠 제약
- 금지 주제
- 민감 정보 처리
- 톤/스타일 제한

## 기술 제약
- 토큰 제한
- 응답 시간
- 형식 제한

## 법적 제약
- 저작권
- 개인정보
- 규정 준수

## 비즈니스 제약
- 브랜드 가이드
- 경쟁사 언급 금지
- 가격/약속 제한
```

---

### 8. 기존 예시

**질문**:
- "참고할 수 있는 기존 프롬프트가 있나요?"
- "원하는 결과물의 예시가 있나요?"

**활용 방법**:
- 좋은 예시 → 성공 기준 도출
- 나쁜 예시 → 안티패턴 파악
- 형식 예시 → 출력 형식 결정

---

## Claude 4.x 확장 수집

### 9. 멀티스텝 여부 (STATE_TRACKING)

**질문**:
- "이 작업이 여러 단계로 나뉘나요?"
- "중간에 중단되어도 이어서 할 수 있어야 하나요?"

**판단 기준**:
```markdown
## STATE_TRACKING 필요 (하나라도 해당)
- [ ] 3단계 이상 순차 작업
- [ ] 10개 이상 항목 순차 처리
- [ ] 장시간 소요 (세션 중단 가능)
- [ ] 체크포인트/롤백 필요
```

---

### 10. 도구 사용 여부 (TOOL_USAGE)

**질문**:
- "파일을 읽거나 써야 하나요?"
- "외부 명령을 실행해야 하나요?"
- "웹 검색이나 API 호출이 필요한가요?"

**판단 기준**:
```markdown
## TOOL_USAGE 필요 (하나라도 해당)
- [ ] 파일 시스템 접근 (읽기/쓰기)
- [ ] 외부 명령 실행 (Bash)
- [ ] 웹 검색/페이지 fetch
- [ ] 외부 API 호출
```
